Classes explained

Why use a class?

When creating a game, you will probably have the same object appear many times and even at the same time. In Tetris, for example, you have many of the same shapes with varying colors. Just like with our ball, we could copy and paste the same code every time we wish to create another shape, but this would be very inefficient and bug prone.

Imagine creating a Tetris clone and deciding you want the shapes a little larger. You would then have to change the size of every single shape, and every color variation of that shape. If you instead used a class, you would only have to change the size in one place for each shape (or even once for every single shape). This is why classes are such an important concept in game development.

The syntax

Classes may look a little daunting at first. Let’s take our Ball class from last week and break it down. Here’s the code again for reference:

class Ball():

    def __init__(self, color, x, y, radius, speed):

        self.color = color

        self.rect = pygame.Rect(x, y, radius * 2, radius * 2)

        self.speed = [speed, speed]

    def update(self):

        self.rect = self.rect.move(self.speed)

        if self.rect.right >= DISPLAYSURF.get_width() or self.rect.left <= 0:

            self.speed[0] = -self.speed[0]

        if self.rect.top <= 0 or self.rect.bottom >= DISPLAYSURF.get_height():

            self.speed[1] = -self.speed[1]

    def draw(self):

        pygame.draw.circle(DISPLAYSURF, self.color, self.rect.center, self.rect.width / 2)

The code starts out with class Ball():, which simply tells Python we’re going to be creating a class named Ball.

The next line is interesting because the function name starts and ends with two underscores. This is because __init__ is a reserved function name in Python. This means we can only use it for its intended purpose and not as a generic name. This special init function is executed every time we create a new instance of this class. Meaning that whenever we use this class, the init function will be called automatically, instead of us manually having to type ball.__init__().

You can see the init function takes parameters, just like the pygame.draw functions. These parameters are passed in when creating a new instance of this class, and can be accessed only within the init function. This is why we create other variables with the same value, so that we may use them later.

Another new keyword is “self”. Self, in a sense, stores the identity of each ball class instance. Just like how everyone has a name, but you have a specific name, self.color stores the color of that specific instance of the class. Each class has a color assigned to self.color, but self refers to the specific instance you’re calling it from. If I made a ball1 instance of the Ball class and instantiated it with “ball1 = Ball(blue)”, then got the value of “ball1.color”, it would return “blue”. This is because when creating our class, we refer to any name the instance takes on as “self”.

The rest of the “init” function looks pretty familiar. We take the parameters and set them equal to variables much like we did before using classes. The update and draw functions look almost identical to the code we used before classes. The only difference is replacing our specific variable names of “ball” or “box” to the variables on “self”.

FPS

If you remember from Week 1, we learned that our monitors clear and draw a new “frame” 60 times per second. These frames are simply pictures of our current game state. And in order to create smooth motion, we change how our game looks a little bit every frame. When these frames are rapidly displayed (at 60 frames per second) it looks as though our objects are moving on the screen.

Up to this point, we haven’t been enforcing our game to run at 60 frames a second. In fact, your games have likely been running at 100 - 200+ FPS! While it may seem like a good thing our games are  running so fast, it actually degrades the user experience. Because our game is producing 200 frames a second but our monitors can only display 60, we miss many of these frames, resulting in janky motion and screen tearing.

We could manually limit the frame rate by sleeping a set amount of milliseconds after every frame (using time.sleep()), but this would lead to the FPS varying wildly from computer to computer. Instead we can use a built-in function of pygame that will take the FPS we want and sleep only the required amount of time to achieve it. And even better, it only requires us to add three lines of code to our game. Add in these lines where they appear:

WHITE = (255, 255, 255)

# Draw screen

myball.draw()

You’ll instantly notice that your game runs much slower now. This is a good thing, as we’ll see every frame our game creates and it will run at the same speed regardless of our machine’s speed. You may need to increase the speed of your balls to match how fast they previously moved.

Vector physics

In order to bounce our ball(s) around the screen, we currently reverse the speed in the appropriate dimension. This works well for our use case, but there’s a better way to go about it, and it involves some math. Before we get to the code, let’s have a short math review.

Vectors are defined as “a quantity having direction as well as magnitude”. This means instead of storing both our speed and direction in a single variable, we will store a vector with magnitude (speed) and direction (angle). This will also allow us to set the exact angle we want our ball to travel at, instead of a always traveling in  a perfectly diagonal line. Here’s a visual demonstration where Z is the speed.

In the first picture, we can see the angle and speed relative to a graph. Then in the second, we see how to get the individual x and y speed from the vector with a little bit of math. The “d” in dx and dy stands for delta. In math, delta is defined as “a finite increment in a variable”. Or in our terms, “the increment in our objects x and y positions”. In fact, using vector physics is really nothing more than controlling the speed and angle, then using math to convert into a speed for each dimension, instead of directly controlling the speed of each direction.

We’ll have to change a couple areas in our Ball class to use vector physics. We’ll use radians when calculating the deltas, but to make it easier to visualize, we’ll take in an angle of degrees. You’ll have to change your code to match the underlined portions below.

Exercise #2 - ball-class.py

import pygame, sys, math

from pygame.locals import *

...

# Create the ball class

class Ball():

    # This code gets executed as soon as we create a new instance

    def __init__(self, color, x, y, radius, speed, angle=-45):

        self.color = color

        self.rect = pygame.Rect(x, y, radius * 2, radius * 2)

        self.speed = speed

        self.angle = math.radians(-angle)

    # Update our game state by moving and bouncing if needed

    def update(self):

        delta_x = self.speed*math.cos(self.angle)

        delta_y = self.speed*math.sin(self.angle)

        self.rect = self.rect.move(delta_x, delta_y)

        if self.rect.right >= DISPLAYSURF.get_width() or self.rect.left <= 0:

            self.angle = math.pi - self.angle

        if self.rect.top <= 0 or self.rect.bottom >= DISPLAYSURF.get_height():

            self.angle = -self.angle

This code isn’t anything too crazy. First we import the math module. Next, we add an optional angle variable, which we default to -45 degrees. Then we set the speed to a single variable, instead of a list. Following that, we get the angle and convert from degrees to radians. Next, we calculate the the x and y deltas, and move our ball with those values. Finally, we replace our reverse x and y bounce logic with proper angle adjustments for bouncing in each dimension.

When you try this code, you may notice that nothing actually changed in your game. This is good because it means that our vector physics method works as intended. However, there is one addition. We can now set the angle when creating our ball instance by adding an extra parameter. Go ahead and try setting different angles. Try to get your ball to travel in just one dimension, then both again.

Ball collisions

Using a list

Our game is starting to get somewhere. We now have a ball class where we can easily create balls of varying sizes, colors, speeds, and directions. But wouldn’t it be a lot more interesting if all these balls collided with each other? Let’s do just that.

Before we can actually check which balls are colliding, we need to keep track of all the balls in our game. If you completed last week’s challenge problem, then you already have a head start on this task. First, we need to make an empty list we can use to store the balls in. Then we’ll use a for loop to populate our list with balls in different locations. Here’s what that code would look like:

Collision handling

We want to make our balls bounce off each other, so we need to check for collisions, then bounce in the opposite direction. We’ll iterate over our ball_list with a simple for loop and use pygame’s rect.colliderect() function to check for collisions. Add this highlighted code to the bottom of your update function:

Exercise #4 - ball-class.py

# Update our game state by moving and bouncing if needed

def update(self):

    delta_x = self.speed*math.cos(self.angle)

    delta_y = self.speed*math.sin(self.angle)

    self.rect = self.rect.move(delta_x, delta_y)

    if self.rect.right >= DISPLAYSURF.get_width() or self.rect.left <= 0:

        self.angle = math.pi - self.angle

    if self.rect.top <= 0 or self.rect.bottom >= DISPLAYSURF.get_height():

        self.angle = -self.angle

    for ball in ball_list:

        if self.rect.colliderect(ball.rect):

            self.angle = self.angle - math.pi

Running this code, we can see we don’t get what we expected. All the balls just vibrate back and forth instead of moving around and bouncing. Do you know why this is happening? If we think about it, we’re checking if our ball collides with every ball in our list of balls. Well what is in this list of balls besides the other balls we’re checking against? The balling we’re currently checking with. This means that this code will always detect our ball as colliding, because it’s comparing against itself. Let’s add an extra condition to factor this out.

Exercise #5 - ball-class.py

for ball in ball_list:

    if not ball is self and self.rect.colliderect(ball.rect):

        self.angle = self.angle - math.pi

Great! Our balls are working as intended… mostly. You can see that they sometimes get stuck on the edge, much like the random bounce behavior you saw last week. We need to check if our ball is in the middle of bouncing off the edge and ignore a ball collision if so. This isn’t the most realistic way of handing this bug, but it is the simplest.

Exercise #6 - ball-class.py

def update(self):

    delta_x = self.speed*math.cos(self.angle)

    delta_y = self.speed*math.sin(self.angle)

    self.rect = self.rect.move(delta_x, delta_y)

    hit_bounds = False

    if self.rect.right >= DISPLAYSURF.get_width() or self.rect.left <= 0:

        hit_bounds = True

        self.angle = math.pi - self.angle

    if self.rect.top <= 0 or self.rect.bottom >= DISPLAYSURF.get_height():

        hit_bounds = True

        self.angle = -self.angle

    for ball in ball_list:

        if not hit_bounds and not ball is self and self.rect.colliderect(ball.rect):

            self.angle = self.angle - math.pi

As you can see, we added a variable that will default to False, and change to True if the ball has collided with the outside bounds of the window. If hit_bounds equal True, then we just ignore the ball collision and bounce off the wall.

If you begin cranking up the amount of balls, you may notice another bug. The balls sometimes get “stuck” in each other, and don’t separate for a little while. We can solve this problem too, by ignoring that collision. Again, this is not the most accurate way of solving the problem, but it works as a simple solution for us. For this, we’ll use a break statement. Whenever you use break within a for loop, it will instantly stop the loop, instead of iterating over the remaining elements. This means that each ball will only collide with up to one other ball per frame.

Exercise #7 - ball-class.py

for ball in ball_list:

    if not hit_bounds and not ball is self and self.rect.colliderect(ball.rect):

        self.angle = self.angle - math.pi

        break
