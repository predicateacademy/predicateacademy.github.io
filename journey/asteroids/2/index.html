Consulting the docs

A very common practice when learning a new programming language, library, or technology in general, is to consult the documentation. The documentation (or docs for short) can be thought of as an encyclopedia created for that specific language, library, etc..  The documentation provides details on every feature and how we can use it. If we wanted to draw a shape using PyGame, but weren’t sure how, we could consult the docs.

Open the PyGame docs by navigating to http://www.pygame.org/docs. Now looking at the top of the page, you can see there’s a “draw” link amongst all the other documented features. If you click on it, you’ll get a full description of the draw module along with all the functions it offers. Let’s take the first function, pygame.draw.rect(), and use it in our code.

import pygame, sys

from pygame.locals import *

pygame.init()

DISPLAYSURF = pygame.display.set_mode((640, 480))

pygame.display.set_caption('Drawing with PyGame')

while True:

    for event in pygame.event.get():

        if event.type == QUIT:

            pygame.quit()

            sys.exit()

    pygame.draw.rect()

    pygame.display.update()

This code is the same as last week, with the addition of the draw.rect() function. If we try to run this code, we’ll get a TypeError: function takes at least 3 arguments (0 given). Let’s refer back to the documentation to figure out what’s causing this error.

Looking under the pygame.draw.rect() heading this time, we can see: rect(Surface, color, Rect, width=0) -> Rect. What does that line mean? Let’s discuss the portion in parenthesis first. This function takes four comma separated values, known as parameters. Parameters are pieces of data or variables that we “pass into” the function. The first parameter we must pass in is a surface. We only have one surface in our program, the display surface, so let’s pass that in. Our updated draw call now looks like this:

pygame.draw.rect(DISPLAYSURF)

The next parameter is color. We’ll go over using different colors shortly, but for now let’s use white, which equates to (255, 255, 255). We could insert this directly into our function call like this:  pygame.draw.rect(DISPLAYSURF, (255, 255, 255)). But as you can see, our line is already getting pretty long  and we still have 2 parameters to go. So instead let’s make a color variable that we can reuse with the value for white. Add in the underlined portion to your code:

color = (255, 255, 255)

pygame.draw.rect(DISPLAYSURF, color)

Next is the Rect parameter. A Rect is a tuple of 4 numbers, the left x position, top y position, width, and height. Here is an illustration of a Rect, where the black border is your window and the black box is your rectangle.

Let’s add a 100 x 50 rectangle at the x/y position (50, 100) into our code.

color = (255, 255, 255)

rect = (50, 100, 100, 50)

pygame.draw.rect(DISPLAYSURF, color, rect)

You may notice in the documentation that the last parameter has an equals sign and value of 0 (width=0). This denotes that the parameter is optional. If we don’t explicitly pass in this parameter, it will be equal to its default value, which is zero in this case. Because of this, let’s ignore width for now.

The final part of the documentation line is this arrow pointing to Rect (-> Rect). This means that the function will return a Rect object, much like the one we just created. In order to actually use the Rect this function returns, we would have to set a variable to the function call. But again, we won’t worry about this for now.

Now that we have a basic understanding of reading and applying the documentation, go ahead and run your code. You should see a white rectangle 50 pixels from the left of your window and 100 pixels from the top that is 100 pixels in width and 50 in height. Congratulations, you just created your first drawing with PyGame! Try creating a circle, ellipse, and line with PyGame by reading more of the draw documentation. When drawing these shapes, you’ll want to put the coordinates tuple directly in your draw call, instead of using the rect variable we created.

RGB Colors

As you’ve learned, every pixel produces red, green, and blue light in order to create colors on your screen. We use this same red, green, and blue (RGB) concept to create our own colors. In Python, we use tuples to store these color values in the same order of (red, green, blue). Each color can have a value from 0, meaning none of the color is being shown, to 255, which means all of the color is being shown. For example, complete blue would be (0, 0, 255), completely green would be (0, 255, 0), and red would be (255, 0, 0).

Try making your shapes purple by changing the color variable in your code. Remember, purple is a mix of red and blue. Then experiment yourself and try to make as many colors as you can.

Constants

Once you’ve discovered some colors you like, make variables for those colors with all uppercase names. For instance, I would make a variable for navy blue like so:

NAVYBLUE = (0, 0, 128)

You may notice that we also use all uppercase characters for our DISPLAYSURF variable. These are called constants. Constants are variables that won’t be changed in the future. Once we set a color to a specific value or the surface we’ll use for the display, we won’t be changing it. We name them in all capitalizes to help us better identify these constants from other variables. This naming scheme is not required by Python, but is highly recommended to aid both ourselves and others looking at our code.

Moving a shape

Now that we’ve made some shapes, found some colors, and learned about constants, let’s make shapes move around. Alter your code or create a new file with the underlined sections below. NOTE: All underlined variables with spaces such as box_x and x_speed contain underscores (_) that are hidden by the underline.

import pygame, sys

from pygame.locals import *

pygame.init()

DISPLAYSURF = pygame.display.set_mode((640, 480))

pygame.display.set_caption('Moving Box')

# Set our color constants

BLACK = (0, 0, 0)

WHITE = (255, 255, 255)

# Set the box properties

box_x = 0

box_y = 200

box_width = 100

box_height = 50

x_speed = 1

# Game loop

while True:

    # Handle events

    for event in pygame.event.get():

        if event.type == QUIT:

            pygame.quit()

            sys.exit()

    # Update game state

    box_x += x_speed

    # Draw screen

    DISPLAYSURF.fill(BLACK)

    pygame.draw.rect(DISPLAYSURF, WHITE, (box_x, box_y, box_width, box_height))

    pygame.display.update()

Now, the only new code here is box_x += x_speed and DISPLAYSURF.fill(BLACK). The former simply increases our box’s position by x_speed every frame. The latter re-draws a black background every frame for the same reason we needed to re-draw the background in our 0s and 1 example last week.

If we run the code, we can see a white box travel across the screen, but never return. This is a start for moving objects, but wouldn’t it be more interesting if it bounced back and forth? Before we start writing code, let’s figure out the most efficient way to bounce our rectangle.

Bouncing

The first thing we need to work out is how to get the object travelling in the opposite direction. We are currently moving the object to the right by increasing its x position with our x_speed variable. If we recall a cartesian coordinate plane, we move to the right by increasing our x, and move to the left by decreasing our x. Since we’re using a speed variable to handle adding to our x position, we could check if our box was outside the window bounds and subtract, instead of adding the speed. Right?

Thinking through this solution, we can quickly realize why it won’t work. If we only subtract from our x position when the box is entirely out of bounds, then as soon as the box has moved even slightly to the left, it will begin moving right again. This will cause a continuous loop of the box getting “stuck” on the edge, instead of bouncing off it.

What we need to do is “toggle” the box’s direction instead of temporarily changing it. This will allow the box to hit the edge, toggle direction to moving left, and continue all the way through the bounce. Since we know we set the speed to 1, we could simply change the speed to -1. This would solve our problem, but also creates another problem.

If we were to write this bounce code and then change our speed variable, it would only change the speed of the box moving to the right. This is because as soon as we bounce, we explicitly set the speed to -1, meaning our box will move left at a speed of 1. We used a variable for speed so that we wouldn’t have to duplicate code changes. But with this alteration we must change the speed in two places in our code.

Luckily for us, Python is smart enough where we can use the variable in our statement for setting a new value. This means instead of setting x_speed to -1, we can set it to -x_speed, which would be equivalent to -1 when x_speed equals 1. This also means we can change x_speed in our program and the bounce code will use our new value with no extra changes!

We just saved ourselves a lot of time and possibly headache from debugging and fixing our code, just by thinking it through beforehand. Now that we’ve worked out a solution, let’s write the code. Add in the underlined code below to your file.

# Update game state

box_x += x_speed

if box_x >= DISPLAYSURF.get_width():

    x_speed = -x_speed

Running the code now, we can see the box bounces off the edge as expected, but then travels out of the window bounds in the other direction. Let’s check to see if the box has gone too far to the left and reverse direction again if it has. Add the underlined portion.

if box_x >= DISPLAYSURF.get_width() or box_x <= 0:

        x_speed = -x_speed

We now get a bouncing box as expected! However, you may notice that the box bounces off the right edge after the left side of the box passes, instead of the right. To fix this we’ll have to keep track of the left side + the width, in order to get the right coordinates of our box. Luckily, PyGame has some built in functions to do this for us, so let’s use them instead.

# Set the box properties

box = pygame.Rect(0, 200, 100, 50)

speed = [1, 0]

# Game loop

while True:

    # Handle events

    for event in pygame.event.get():

        if event.type == QUIT:

            pygame.quit()

            sys.exit()

    # Update game state

    box = box.move(speed)

    if box.right >= DISPLAYSURF.get_width() or box.left <= 0:

        speed[0] = -speed[0]

    # Draw screen

    DISPLAYSURF.fill(BLACK)

    pygame.draw.rect(DISPLAYSURF, WHITE, box)

    pygame.display.update()

Here we created a PyGame Rect object. But instead of creating a tuple with our values like before, we used the syntax pygame.Rect(). This is a little bit of “magical” code in which PyGame takes care of creating a tuple for us, along with keeping track of the center, top left, and more handy attributes of our rectangle. More information, including all the attributes a PyGame Rect contains can be found in the documentation at http://www.pygame.org/docs/ref/rect.

As you can see, the rectangle now properly bounces back and forth between the window borders.

To start, you’ll want to draw a circle instead of a rectangle. Keep the pygame.Rect as your ball, you’ll still need it for calculating collisions and moving the ball. You’ll want to use the center of the ball in your draw call, instead of the top left x and y positions. The PyGame Rect object has a .center property to keep track of this for you.

When drawing your circle, remember the draw call for a circle wants the radius, not width. If you’re using the Rect’s width, you’ll need to divide by 2.

Apart from colliding with the left and right, you’ll have to add a similar if statement to check for top or bottom collisions and reverse the correlating speed. The first speed attribute (speed[0]) is for the x speed, and the second (speed[1]) for y.
