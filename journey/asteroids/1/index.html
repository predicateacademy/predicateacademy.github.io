Intro

How do we make games with Python?

SDL

When making a video game there are a lot of variables to consider. What platforms are you going to create the game for? Just Windows? Are you going to use OpenGL or DirectX to take advantage of a graphics card? Or how about if you want your game to also run on macOS and Linux? And what if you want to sell your game on mobile operating systems like Android and iOS?

To help solve many of these issues, some people developed a programming library that handles these “low-level” problems with making a game. This library is called Simple DirectMedia Layer, or SDL for short. SDL allows us to write simpler code without having to worry about the details like what operating system we’re using. You’ve used parts of SDL already for things like playing sounds in Python.

PyGame

Even though we’ve previously used SDL, you probably didn’t notice you were. This is because SDL is written in the C programming language and originally intended to work with C and C++. Some kind folks wanted to make games using SDL in Python, so they created PyGame. PyGame is what’s called a binding for SDL, which allows us to take advantage of this library, made using C, inside our Python code.

Now that you know how we’re going to make a game in Python, let’s go over why we’d want to.

CLI vs GUI

Up to this point at Predicate, we’ve interacted with Python through a CLI or Command Line Interface. CLIs are great for getting input in simple programs that only do one thing at a time. When you get into more complex programs, more like the ones you use everyday, we need more than just text input and output. What we need is a Graphical User Interface, or GUI. When we use a GUI to start adding graphics, images, and mouse input, our program becomes a lot more interesting and functional.

The Basics

Creating a window

Let’s get started with PyGame by creating a window. Type the following code into IDLE.

import pygame, sys

from pygame.locals import *

pygame.init()

DISPLAYSURF = pygame.display.set_mode((640, 480))

pygame.display.set_caption('Hello Predicate!')

while True:

    for event in pygame.event.get():

        if event.type == QUIT:

            pygame.quit()

            sys.exit()

    pygame.display.update()

Run the code and you’ll have your very first window using PyGame. This window is pretty boring though. It’s entirely black with a window title of “Hello Predicate!”.

Now there’s a bit of code we’ve never seen before, so let’s go over it. The first unfamiliar line here should be pygame.init(). This is similar to the pygame.mixer.init() we’ve used previously and simply initializes everything in PyGame so we can use it in our code.

Next is the line where we set DISPLAYSURF, which is our variable for the display surface. A surface is a very important idea in PyGame. You can think of a surface as a blank canvas where we can draw lines, circles, or rectangles, fill with color, or place images. We can also copy to and from surfaces. You’ll have many surfaces when making a game, but there is one that’s special, which is your display surface. The display surface is what we’ll actually see on the screen, therefore we can only have one of them.

The next line in our code sets the caption, or title, of our PyGame window.

Then we get to a more interesting line, the for loop. This loop will go over every event in pygame.event.get() and allow us to act on it. In this case we just check if the event type is QUIT and exit the program if it’s true.

The last line, pygame.display.update(), draws all the changes that we’ve made to the display surface to the window. Since we haven’t drawn anything to the surface, our window is completely blank.

Game loop

Let’s go back to events for a moment. An event is any action that the user does inside the window. An event could be moving the mouse, clicking, or pressing a key, amongst other actions. We can get a list of events that have happened by calling pygame.event.get(). Every time we get the events in this list, it clears itself so that we don’t deal with the same event twice. As you can see from our code, every event has a type property which tells us if the action was a mouse click, key press, quit, etc..

Apart from handling events in our while True loop, we also draw the display surface to the screen. These are the two main things that happen in our while True. The last thing we’ll do in this loop is update our game state. The game state is what we call all the variables that keep track of our game. For instance, you may have a player health or score keeping variable in your game, which would be considered part of your game state. Since nothing is happening in our code yet, we don’t have any state to keep track of or update.

The while True loop is what we call our game loop. The game loop is simply the main loop that we continue running in our game that controls events, updates the game state, and draws to the screen. Here’s a visual representation:

Pixels

Now that we have a broader overview of how a game works, let’s dive into actually drawing to the screen. If we’re going to be drawing to a screen, we should probably understand how a screen works.

Every screen contains a large array of colored dots, known as pixels. You’ve probably heard of pixels before, but what exactly is a pixel? A pixel is an individual dot that mixes red, green, and blue light in order to produce color. It you looked at your computer monitor with a microscope, you could see a pixel that looks something like:

When you take many of these pixels (over 2 million in the screen in front of you) and put them side by side in a display, you start to get a viewable image.

You may have heard terms such as “Full HD” or “1080p” before, both of which refer to how many pixels the display contains. How many pixels your screen contains is measured in width by height (i.e. 1280 x 720), and known as resolution. To get a list of resolutions that your screen supports, add the underlined code to your file:

pygame.display.set_caption('Hello Predicate!')

print pygame.display.list_modes()

while True:

The largest combination of numbers (i.e. 1920, 1080) is how many pixels are in your screen. In order to make motion look smooth, your monitor changes these pixels 60 times a second, or has a refresh rate of 60 Hz. We call drawing to the monitor a frame, so your monitor has 60 frames drawn to it every second. When an object is moving across your screen, it’s really just being drawn a couple pixels away every single frame. When this happens very quickly (60 times a second), it looks like smooth motion to our eyes.

Moving pixels

Alright, we understand how a game works and how a screen works, let’s make a character move. But before we move him in our window, let’s move him in our terminal to get a better understanding of drawing pixels. Type the following code into IDLE.

screen = [0, 0, 0, 0, 0, 0, 0, 0] # Simulate an 8 x 1 screen

screen_w = len(screen) # Set screen_w to the screen width

background = [0]*screen_w # Make the background as many 0s as the screen

player_pos = 3 # Set the position of the player

for x in range(screen_w): # Copy the background to the screen

    screen[x] = background[x]

screen[player_pos] = 1 # Set a 1 at the player's position

print screen # Draw the screen

With this code we successfully made a background of 0s and a player denoted by a 1 appear in the terminal. What if we want to move the player to the right one space? Let’s add this code to the bottom of our file.

player_pos += 1

screen[player_pos] = 1

print screen

It looks like this code will change our player’s position by one, then draw that change to the screen. If we run the code we’ll see that it’s not the result we wanted. Instead of moving the character one space it simply added the character there. In addition to drawing the player in his new location, we must also replace his old location with background to make it look as if he moved from one location to the other. This is a fundamental concept when drawing graphics, and one that takes a lot of processing power.

To fix this problem, let’s first re-draw our background before moving the player:

for x in range(screen_w):

    screen[x] = background[x]

player_pos += 1

screen[player_pos] = 1

print screen

I copied the same for loop that we used above to draw the background before we move the player. Now we get the result we’re looking for! The player moved one space to the right and the background replaced his previous location, giving the sense of movement.

Now we can see a pattern emerging where we’ll be re-drawing the background and player every frame. Let’s create a short loop and clear the screen every frame with this code:

import os, time

screen = [0, 0, 0, 0, 0, 0, 0, 0]

screen_w = len(screen)

background = [0]*screen_w

player_pos = 3

for i in range(4):

    for x in range(screen_w):

        screen[x] = background[x]

    screen[player_pos] = 1

    os.system('clear')

    print screen

    time.sleep(0.5)

    player_pos += 1

WIth this code, we have a basic drawing animation that occurs over 4 frames. There’s no input to deal with here, but we can see two parts of the game loop in action. We update our player’s position every frame, so we’re updating our game state. Then we draw the background and player to the screen, that’s the drawing part of our game loop. You can imagine this in a while True loop happening every frame of your game while also dealing with user input. And there you have it, a basic game loop.
