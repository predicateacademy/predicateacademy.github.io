Using sprites

A good first step when learning a new feature in a language/library is to consult the documentation. Let’s head to http://www.pygame.org/docs/ref/sprite to see how the sprite module works.

If we look at the section for “pygame.sprite.Sprite”, we can see it tells us that this is a Simple base class for visible game objects. Underneath it lists some functions of the Sprite class, none of which we’ll be using right now. Then we have an example Block class using this PyGame Sprite. There are some nice comments on the example class explaining how it works. Let’s start converting our Ball class to look more like this example.

Exercise #2 - ball-sprite.py

# Create the ball class

class Ball(pygame.sprite.Sprite):

    # This code gets executed as soon as we create a new instance

    def __init__(self, sprite, x, y, radius, speed, angle=-45):

        # Initiate this sprite

        pygame.sprite.Sprite.__init__(self)

        # Setup our sprite surface

        self.image = pygame.image.load('sprites/' + sprite)

        self.image = pygame.transform.smoothscale(self.image, (radius * 2, radius * 2))

        # Get our rectangle and set the position

        self.rect = self.image.get_rect()

        self.rect.topleft = (x, y)

        # Create our vector variables

        self.speed = speed

        self.angle = math.radians(-angle)

Let’s walk through this new code a little bit. First we add “pygame.sprite.Sprite” into our Ball class. This may look like we’re taking in Sprite as a parameter, but remember that we take parameters in the __init__ function. Instead, this is telling Python that we want to base our class on the “pygame.sprite.Sprite” class. This aligns with what we read in the documentation about Sprite being a “Simple base class for visible game objects”. Note that we also change our color parameter to “sprite” for accuracy.

Next, we use the same init line from the documentation. This will call the init function of the parent class (Sprite in this case).

We also set self.image, similar to the documentation, but instead of using a color we load an image. On the second self.image line, we scale the image to the radius size we gave as a parameter. If we did not resize the image, then it would use the same dimensions as the actual image file. It is much easier to alter this size in code than to change the image’s dimensions each time, so we use “smoothscale” to do so.

Lastly, we get the image’s rectangle, just like in the documentation. This is a PyGame Rect, just like we used before, but generated by the size of our image. We also add a second line where we set the position of the rectangle to the x and y parameters.

We’ve now implemented sprites into our Ball class. The last thing we need to do is draw the sprite instead of a circle. We’ll do this by blitting it to the display surface. Up to this point we’ve been drawing directly to our display surface and haven’t needed to “blit” any surface. Blitting is the act of copying one surface onto another. Since we will only ever see what is on the display surface, we have to copy our sprite to that surface.

Here, we blit our Sprite and update our ball_list with a sprite replacing color.

Exercise #3 - ball-sprite.py

    # Draw our ball to the screen

    def draw(self):

        DISPLAYSURF.blit(self.image, self.rect)

ball_list = []

# Create a list of 10 Ball instances

Pixel format

You may notice your game running slow with many sprites bouncing around your screen. This is because of your sprite’s pixel format. The pixel format is how images store their color, depth, etc. values. Most of the time, the pixel format of your image and display are going to differ, so you’ll have to convert. Right now, PyGame is converting the pixel format of our ball sprites to the pixel format of our display every single frame. To fix this, we’ll use .convert() when loading our image in PyGame, which will convert it once before we ever need to draw it. Let’s add that to our code.

Exercise #4 - ball-sprite.py

def __init__(self, sprite, x, y, radius, speed, angle=-45):

        # Initiate this sprite

        pygame.sprite.Sprite.__init__(self)

        # Setup our sprite surface

        self.image = pygame.image.load('sprites/' + sprite).convert()

        self.image = pygame.transform.smoothscale(self.image, (radius * 2, radius * 2))

Running this code now, you’ll notice a white outline around your sprites. This is because it’s not converting your images with their alpha values. The alpha value is how transparent or opaque your image appears. An image that is fully transparent will be invisible and only the background behind it will show. While a completely opaque image will show only the image without appearing “see through”.

To preserve this alpha value and make the edges of your image appear transparent, we’ll use the .convert_alpha() function.

Exercise #5 - ball-sprite.py

self.image = pygame.image.load('sprites/' + sprite).convert_alpha()

Sprite groups

Now that you know how to load and draw a sprite in PyGame, let’s use a PyGame Sprite Group to manage our list of balls and provide some extra functionality. You can view the documentation for Sprite Groups at pygame.org/docs/ref/sprite#pygame.sprite.Group.

As can be seen in the documentation, Group is “A container class to hold and manage multiple Sprite objects”. Instead of using a Python list to store our balls, we’ll use a PyGame Group. This will eliminate the need for our update and draw loops, as Group includes that functionality. Here are the code changes:

Exercise #6 - ball-sprite.py

# Make our list variable

ball_list = pygame.sprite.Group()

# Create a list of 10 Ball instances

for i in range(10):

    ball_list.add(Ball('ball.png', i * 40, i * 40, 20, 5))

# Game loop

while True:

    # Handle events

    for event in pygame.event.get():

        if event.type == QUIT:

            pygame.quit()

            sys.exit()

    # Update game state

    ball_list.update()

    # Draw screen

    DISPLAYSURF.fill(BLACK)

    ball_list.draw(DISPLAYSURF)

    pygame.display.update()

As you can see, this change results in slightly less code. There is also another advantage Group provides, only clearing the area of screen that has changed. Currently, our game redraws the entire background every single frame. While it is required to redraw the background to keep our objects from duplicating, we don’t need to redraw the areas that haven’t changed.

If we were to fix this ourselves, we would have to keep track of where each ball’s last position was, and clear that area before drawing the ball’s new position. Lucky for us, Group has a .clear() function that will keep track of this data “automagically”.

In order to use this feature, we’ll need to create a background surface. Then we can replace our .fill() with .clear() and pass in our background.

Exercise #7 - ball-sprite.py

# Create our background surface

background = pygame.Surface((DISPLAYSURF.get_width(), DISPLAYSURF.get_height())).convert()

# Fill the entire window with background when the game starts

DISPLAYSURF.blit(background, background.get_rect())

# Game loop

while True:

    # Handle events

    for event in pygame.event.get():

        if event.type == QUIT:

            pygame.quit()

            sys.exit()

    # Update game state

    ball_list.update()

    # Draw screen

    ball_list.clear(DISPLAYSURF, background)

    ball_list.draw(DISPLAYSURF)

    pygame.display.update()

    # Only output our desired frame rate

    clock.tick(FPS)

You may notice here that we explicitly blit the background, even though our Group should take care of it for us. This is because Group will only redraw the areas that have changed, meaning the background won’t be entirely drawn when the game starts. We solve this by blitting it once before the game loop begins.

Now that we’re using a Surface for our background, try to use an image instead of solid black. You’ll probably want to download an image that is close to the same dimensions of your game, but remember you can use PyGame to scale the image.
