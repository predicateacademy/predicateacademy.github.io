Levels

We’re going to start by adding levels to our game. This will allow us to shoot all the balls on screen and have more spawn. You can make the levels progressively more difficult, or even completely random.

Exercise #1 - http://pastebin.com/KQ0Ern9h

In this exercise we first import a “levels” module. This module doesn’t exist in the Python library, but just like we can create custom functions and classes, we can also create our own modules. We’re separating out this file to enforce “separation of concerns”.

Next, we set the current_level to 0 and add some code to our main loop. In the main loop we check if there are any Balls still being drawn, and set level_complete to True if none are found. If the level_complete variable is True, we increment the level, print the new level to the console, and spawn balls according to the parameters in our levels list. You’ll want to make sure you remove the previous ball creation loop in your code.

Exercise #2 - http://pastebin.com/R5kdqLhK

Here we create a new file named “levels.py”. In this file we create a levels list with dictionaries. Each dictionary corresponds to a level of your game. All of the parameters can be tweaked and additional levels added.

Lives

To make playing through the levels in your game more interesting, we’re going to add in player lives. You will start the game with a designated number of lives and every time a ball hits your player Ship, you will lose a life. When all of your lives are lost, the game ends.

Exercise #3 - http://pastebin.com/tarwJSRL

We created two new variables in this exercise. The first, game_over, we set to False, and later set to True if all of the player’s lives have been depleted. Then in the main game loop we check if game_over is True. If so, we end the game with a “GAME OVER” message. If not, we update all the sprites, thus continuing the game. The second variable, lives, simply keeps track of how many lives the Ship has left.

The rest of this code loops through the balls and checks for a collision with the player. If a collision is found, the ship loses a life and the ball is destroyed. Make sure you remove the original “sprite.update()” from your code when completing this exercise.

Movement

Our game is starting to come together now, but it would be a lot more interesting if we could shoot in more than just one direction. We’ll solve this problem by adding the ability for our Ship to turn.

Exercise #4 - http://pastebin.com/e3zyLhCV

In this exercise we start by creating a source_image variable. This is a copy of the original state of our ship which we’ll use a reference for any rotations performed. Next we create an angle variable and set it to 90. Then we replace all of our move code with calls to class methods we create. On the “sprites.add” line, we change the predefined angle of 90 to our ships new self.angle variables.

Then we get to the turn and move methods. In turn, we take the current center of the ship and store it. Then we increase the angle of the ship, given the self.speed variable and the multiplier parameter. The multiplier will allow us to set the direction and increase/decrease the turn speed relative to the ship’s speed. Finally, in the move method we use almost duplicate code from the Ball class move method. This will use the ship’s speed, angle, and passed in multiplier to calculate the required movement.
