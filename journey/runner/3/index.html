DOM

In the first week we covered JavaScript’s purpose on the web, which is to provide all logic on the web page. We can use JS to manipulate the webpage by using the DOM, or Document Object Model. That may sound like an intimidating name, but you can think of it as the HTML itself. Opening Chrome’s Developer Tools (F12 > Elements), you can see that your web page is laid out like a tree, with each parent element expanding to reveal it’s children.

Let’s get started with the DOM by adding a button to our page and manipulating it with JavaScript.

Exercise #3 - http://pastebin.com/X02muCsc

The only thing we add to our HTML here is a button element with an id of “button1”. IDs are nearly identical to classes, with the difference being that an ID must always be unique, meaning you may only have one of each on your page. This makes IDs not so suitable for CSS, but work well for JS alterations.

Document is a global variable in JS that refers to the DOM, and querySelector is a function on the document which will select an element on the page. On the second line, we make use of the “style” property on all DOM elements to add CSS styles to our button.

Exercise #5 - http://pastebin.com/zdyQJ85n

This exercise is where things get a bit more interesting. We first add a variable named “clicks”, then use the “addEventListener” function on our button. Event listeners in JS will watch for certain event and execute your given callback whenever they occur. In this case we listen for the “click” event and provide an anonymous function to be called, which we’ll get more into later. In our callback, we increment the amount of clicks by one and set the innerHTML (HTML between the opening and closing tag of our button) to a short phrase with the amount of times the button has been clicked.

Functions

Our web page is getting more interesting, but in the latest exercise we repeat ourselves an awful lot. There’s a “rule” in programming called DRY, which means “don’t repeat yourself”. Looking at our code, we can see that we call both createElement and appendChild on every element we make, as well as setting types on a couple. Although it’s not very cumbersome now, imagine an application that does more than just map one input to one output.

One way of reusing our code is with functions.

Exercise #7 - http://pastebin.com/EyR3j8bS

We refactored quite a bit of code in this exercise. If you’re familiar with creating functions in Python, you’ll notice that the process isn’t entirely different in JavaScript. Instead of “def” we use “function” and instead of a colon, we use curly braces to surround our function’s code.

Our function itself, “makeElement”, isn’t the best name in this case, but it’ll work fine for demonstration. You may also notice that we actually increased the total lines of code in our file, but adding just a few more elements to our page would offset the initial “cost” of using this function.

The only new code in this function is the “return” keyword. Typically, a function will take a certain input (element, parent, and type in our case) and give back, or return, an output. Since we’re setting our form (and other variables) equal to something, we need to give them something to equal. It may take a little bit of experimentation to fully grasp this subject.

Fetch

Now that we’ve worked with JavaScript on our web page, let’s incorporate data from other places on the web.

Exercise #8 - http://pastebin.com/J2Qq2ZQL

If you run this exercise, you’ll see that it adds a button to your page that, when clicked, displays a random Chuck Norris joke. So what’s going on in the code?

First, we make a joke variable and set the “className”. If you look in Chrome’s Developer Tools (F12), you’ll see that this adds a “class” attribute to the element with the given class name. I haven’t provided any styles for this class, but you may add it to your CSS file and change the styling.

The next new line is “fetch” with a URL enclosed. Fetch, as the name would imply, fetches data from the provided location. In this case, we’re getting a random joke from the ICNDb API (application programming interface). Fetch needs to know what to do once it retrieves the data though, and that’s where “.then” comes in. Even though all of the “.then”s are on their own line, they are all still attached to fetch; we separate them for readabilities sake.

The first “then” will be called when fetch has finished retrieving the data, and all subsequent “then”s will be called when the previous has finished. In our situation, we call .json() in the first and get the value property in the second. The important thing to note in these is that they don’t include curly braces. This is a shorthand notation for returning a single line of code, because it occurs so often.

Finally, in the last line we do something familiar and set the innerHTML of our joke element to the joke text itself.
